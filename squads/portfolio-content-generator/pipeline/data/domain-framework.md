# Domain Framework: Portfolio Copy para Desenvolvedores

## Framework Operacional

### Fase 1: Coleta de Contexto

1. Ler o foco definido (tipo de conteúdo + projeto alvo)
2. Ler arquivos de localização `src/locales/*.json` — contexto do portifólio atual
3. Ler componentes relevantes — entender a estrutura do site
4. Ler `package.json` — confirmar stack real
5. Identificar o tom atual do portifólio

### Fase 2: Extração de Dados (Pedro Projeto)

Para cada projeto identificado:
- Nome e tipo (fullstack, game, frontend, backend)
- Stack com papel de cada tecnologia
- Funcionalidades implementadas (concretas, não genéricas)
- Decisões técnicas com rationale
- Diferencial ou problema resolvido

### Fase 3: Estruturação do Copy (Carla Copy)

Para **bio**:
1. Aplicar Elevator Pitch Formula ou PAS
2. Incluir especialidade + diferencial + quem se beneficia
3. Tom descontraído mas com credibilidade técnica
4. Máximo 3-4 frases

Para **descrição de projeto**:
1. Aplicar framework STAR
2. Começar pelo problema ou contexto (não pela tecnologia)
3. Destacar a decisão técnica não óbvia
4. Fechar com resultado ou impacto (mesmo que qualitativo)
5. Máximo 4-6 frases por projeto

Para **seção do site**:
1. Identificar o objetivo da seção (o que o recrutador precisa entender)
2. Usar linguagem de identidade, não de lista
3. Tom consistente com o restante do portifólio

### Fase 4: Revisão (Renata Revisao)

1. Checar clareza da proposta de valor (30-second scan test)
2. Verificar presença de evidência de impacto
3. Identificar sinais de escrita júnior
4. Avaliar alinhamento de tom
5. Emitir APPROVE/REJECT/CONDITIONAL APPROVE com feedback acionável

---

## Critérios de Decisão

**Quando usar STAR vs. PAS para projeto:**
- STAR: quando tem resultado mensurável ou decisão técnica clara
- PAS: quando o projeto resolve uma dor de usuário identificável

**Quando usar Elevator Pitch vs. narrativo para bio:**
- Elevator Pitch: portifólio voltado para mercado corporativo/recruiter
- Narrativo: portifólio voltado para comunidade, personal brand, freelance

**Quando incluir stack na descrição:**
- Sempre mencionar tecnologias não óbvias (ex: TypeScript num projeto de jogo)
- Nunca listar todas as dependências — somente as que mostram decisão ou diferencial

**Quando sinalizar INCOMPLETO no brief:**
- Projeto não tem demo funcional
- Não há evidência de problema real resolvido
- Código está visivelmente inacabado
