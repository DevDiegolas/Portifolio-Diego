# Anti-Patterns: Portfolio Copy para Desenvolvedores

## Os 8 Erros Mais Comuns (baseado em análise de 200+ portifólios)

---

### 1. Tech Dump sem Contexto

**O que é:** listar tecnologias sem explicar o problema que resolveram.

**Exemplo ruim:**
> "Built with React, Node.js, MongoDB, Docker, AWS, Redis, GraphQL, Jest..."

**Por que é prejudicial:** recrutadores não precisam de inventário de tecnologias. Precisam entender o que você resolve. Uma lista de 12 tecnologias comunica menos do que uma decisão técnica bem descrita.

**Como corrigir:** para cada tecnologia principal, uma frase de contexto: qual problema ela resolveu naquele projeto específico.

---

### 2. Paixão Vazia

**O que é:** declarações genéricas de entusiasmo sem evidência.

**Exemplo ruim:**
> "Passionate developer who loves creating user-friendly applications and is always eager to learn new technologies."

**Por que é prejudicial:** 90% dos portifólios de júnior têm exatamente essa frase. É ruído. Não diferencia, não convence, não informa.

**Como corrigir:** mostrar paixão através de projetos concretos. "Construí um pathfinder A* do zero no fim de semana porque queria entender o algoritmo por dentro" comunica paixão melhor do que qualquer declaração.

---

### 3. Foco no Dev, não no Avaliador

**O que é:** escrever como se o leitor fosse outro desenvolvedor sênior, quando quem lê frequentemente é um hiring manager ou recruiter não-técnico.

**Exemplo ruim:**
> "Implemented a reactive store using the observer pattern with WeakMap-based subscriptions to avoid memory leaks in long-running SPAs."

**Por que é prejudicial:** recruiter não técnico não consegue avaliar. Mesmo para devs: muita complexidade sem contexto de negócio soa pedante.

**Como corrigir:** começar pelo impacto ou problema de negócio, depois explicar a solução técnica em uma frase acessível.

---

### 4. Sem Resultado

**O que é:** descrever o que foi construído sem mencionar o impacto ou o por quê.

**Exemplo ruim:**
> "Built a dashboard with charts and filters."

**Por que é prejudicial:** todo mundo "built a dashboard." O que torna o seu especial é o que ele resolveu. Sem resultado, é só mais um item na lista.

**Como corrigir:** mesmo sem métricas, dar contexto qualitativo: "para reduzir o tempo de análise da equipe", "que substituiu um processo manual de 3 horas por semana."

---

### 5. Clone de Tutorial sem Diferencial

**O que é:** projetos que são réplicas diretas de tutoriais populares — todo-list, weather app, calculadora — sem nenhum problema real resolvido ou twist técnico.

**Por que é prejudicial:** sinaliza que o dev sabe seguir instruções mas ainda não consegue definir e resolver seus próprios problemas. É um sinal forte de nível júnior.

**Como corrigir:** se o projeto começa como tutorial, precisar adicionar algo genuíno — uma feature não ensinada no tutorial, uma decisão técnica própria, ou resolver um edge case real.

---

### 6. Demo Quebrada ou Inacessível

**O que é:** links que levam a telas em branco, errors de deploy, repositórios privados sem instrução de acesso.

**Por que é prejudicial:** o primeiro pensamento do recrutador é "se não consegue manter o próprio portifólio funcionando, como vai manter meu produto?" É um sinal de falta de atenção ao usuário final.

**Como corrigir:** garantir que todo projeto com link de demo esteja funcional antes de publicar. Alternativa: repositório GitHub bem documentado com README claro.

---

### 7. Quantidade sobre Qualidade

**O que é:** 12+ projetos listados, mas nenhum com descrição, screenshot ou link funcionando.

**Por que é prejudicial:** 12 projetos rasos comunicam menos confiança do que 4 projetos polidos com boa documentação, README, e copy bem escrita.

**Como corrigir:** selecionar os 3-5 projetos mais representativos e investir em apresentá-los bem. O resto pode ser mencionado como "outros projetos" com link para o GitHub.

---

### 8. Adjetivos sem Evidência

**O que é:** usar qualificadores vagos sem dados ou contexto que os sustente.

**Exemplos ruins:**
> "High-performance backend", "scalable architecture", "clean and maintainable code"

**Por que é prejudicial:** qualquer dev pode escrever "high-performance." Sem benchmark, sem contexto de escala, sem dado concreto, é propaganda vazia. Cria ceticismo, não confiança.

**Como corrigir:** substituir por dados concretos. "API com tempo de resposta médio de 12ms em produção" bate "high-performance API" toda vez.
